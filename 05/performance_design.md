## 1. Анализ производительности

В рамках данной лабораторной работы рассматривается сервис `file_storage_service`, требуется добавить механизм кеширования и rate limiting поверх уже реализованного REST API.

### 1.1. Используемые хранилища

В сервисе используются следующие хранилища данных:

| Хранилище  | Назначение                                                           |
| ---------- | -------------------------------------------------------------------- |
| PostgreSQL | пользователи, авторизация, папки, файлы                              |
| MongoDB    | документное хранение папок                                           |
| Redis      | кеширование часто читаемых данных и хранение счётчиков rate limiting |

### 1.2. Часто используемые операции

К наиболее часто используемым операциям относятся endpoint-ы, которые используются при стандартной работе пользователя с файловым сервисом:

| Endpoint                       | Назначение                           | Причина возможной высокой нагрузки                                          |
| ------------------------------ | ------------------------------------ | --------------------------------------------------------------------------- |
| `POST /v1/auth/login`          | авторизация пользователя             | критичный endpoint, может часто вызываться и использоваться для brute-force |
| `GET /v1/folders`              | получение списка папок из PostgreSQL | часто используется при отображении интерфейса сервиса                       |
| `GET /v1/files/by-name/{name}` | получение файла по имени             | чтение данных по пользовательскому запросу                                  |
| `GET /v2/mongo/folders`        | получение списка папок из MongoDB    | часто используется как документный backend для папок                        |

### 1.3. Потенциально медленные операции

Наиболее затратными по времени являются операции, связанные с обращением к базам данных:

* проверка логина и пароля в PostgreSQL при авторизации;
* получение полного списка папок из PostgreSQL;
* получение полного списка папок из MongoDB;
* поиск файла по имени;
* операции, повторяющиеся многократно при чтении данных.

При небольшой нагрузке эти операции выполняются приемлемо, однако при большом количестве повторных запросов они создают лишнюю нагрузку на PostgreSQL и MongoDB и увеличивают общее время ответа сервиса.

### 1.4. Требования к производительности

Для файлового сервиса можно принять следующие ориентиры по производительности:

| Метрика       | Описание                                                            | Целевое значение                                        |
| ------------- | ------------------------------------------------------------------- | ------------------------------------------------------- |
| Response Time | время от отправки запроса до получения ответа                       | до 100–150 мс для чтения, до 150–200 мс для авторизации |
| Throughput    | количество запросов, которые сервис способен обрабатывать в секунду | должно оставаться стабильным при повторяющихся чтениях  |
| Stability     | устойчивость системы к всплескам запросов                           | один клиент не должен перегружать endpoint авторизации  |

### 1.5. Вывод по анализу

Наиболее подходящими кандидатами для кеширования в данном сервисе являются endpoint-ы получения списков папок, так как они относятся к часто читаемым данным и удобны для инвалидации при изменении состояния системы.

Для реализации кеширования выбраны:

* `GET /v1/folders`
* `GET /v2/mongo/folders`

Такой выбор позволяет уменьшить нагрузку как на PostgreSQL, так и на MongoDB.

Для rate limiting выбран endpoint:

* `POST /v1/auth/login`

Это связано с тем, что endpoint авторизации наиболее чувствителен к большому количеству повторных запросов и требует защиты от brute-force и избыточной нагрузки.

## 2. Проектирование стратегии кеширования

Для повышения производительности сервиса было принято решение добавить кеширование поверх уже существующей логики работы с PostgreSQL и MongoDB.
Так как лабораторная работа является продолжением предыдущего проекта, кеширование внедряется в текущую архитектуру без переработки основной бизнес-логики.

### 2.1. Выбор данных для кеширования

В качестве кандидатов для кеширования были выбраны часто читаемые данные, которые:

* регулярно запрашиваются клиентом;
* не изменяются при каждом запросе;
* могут безопасно храниться ограниченное время;
* легко инвалидируются при изменении состояния системы.

Для реализации кеширования были выбраны следующие endpoint-ы:

* `GET /v1/folders`
* `GET /v2/mongo/folders`

### 2.2. Обоснование выбора

Endpoint получения списка папок является одним из основных в файловом сервисе, так как при работе пользователя список папок отображается регулярно. Повторные обращения к этим endpoint-ам без кеширования каждый раз приводят к чтению данных из базы.

Выбор именно этих двух endpoint-ов позволяет:

* уменьшить нагрузку на PostgreSQL;
* уменьшить нагрузку на MongoDB;
* сократить время ответа для повторяющихся запросов;
* продемонстрировать кеширование для двух разных backend-ов в рамках одного сервиса.

### 2.3. Выбранная стратегия кеширования

Для обоих endpoint-ов выбрана стратегия **Cache-Aside**.

Принцип работы:

1. При поступлении запроса сервис сначала проверяет наличие данных в Redis.
2. Если данные найдены, они возвращаются из кеша.
3. Если данных нет, сервис получает данные из базы данных.
4. После этого данные сохраняются в Redis с установленным временем жизни.
5. Ответ возвращается клиенту.

### 2.4. Причины выбора Cache-Aside

Стратегия Cache-Aside выбрана по следующим причинам:

* простая реализация;
* хорошо подходит для учебного проекта;
* не требует изменения логики работы базы данных;
* позволяет гибко контролировать время жизни кеша;
* легко комбинируется с существующими PostgreSQL- и MongoDB-endpoint-ами.

### 2.5. Время жизни кеша (TTL)

Для кешируемых данных устанавливается TTL **60 секунд**.

Такой TTL выбран как компромисс между:

* уменьшением нагрузки на базу данных;
* сохранением достаточно актуальных данных;
* простотой реализации.

Для данной лабораторной работы значение 60 секунд является достаточным, так как список папок изменяется не настолько часто, чтобы требовалась более сложная стратегия обновления.

### 2.6. Ключи кеша

Для хранения данных в Redis используются следующие ключи:

| Ключ                      | Содержимое                 |
| ------------------------- | -------------------------- |
| `cache:folders:pg:all`    | список папок из PostgreSQL |
| `cache:folders:mongo:all` | список папок из MongoDB    |

### 2.7. Инвалидация кеша

Кеш должен сбрасываться при изменении данных, влияющих на содержимое списка папок.

Для PostgreSQL-ручек инвалидация выполняется при:

* `POST /v1/folders`
* `DELETE /v1/folders/{folder_id}`

Для MongoDB-ручек инвалидация выполняется при:

* `POST /v2/mongo/folders`
* `DELETE /v2/mongo/folders/{folder_id}`

После создания или удаления папки соответствующий кеш удаляется, и следующий запрос снова получает актуальные данные из базы.

### 2.8. Ожидаемый эффект

Ожидается, что использование кеширования позволит:

* сократить количество повторных обращений к PostgreSQL и MongoDB;
* уменьшить среднее время ответа для endpoint-ов получения списков папок;
* повысить пропускную способность сервиса;
* уменьшить нагрузку на backend при повторяющихся запросах.

## 3. Реализация кеширования

В системе реализовано кеширование с использованием Redis в качестве in-memory key-value хранилища.

Кеширование применяется для ускорения операций чтения и снижения нагрузки на PostgreSQL и MongoDB.

Используется шаблон Cache-Aside:

1. При поступлении запроса сервис проверяет наличие данных в Redis.
2. Если данные найдены, они возвращаются из кеша.
3. Если данные отсутствуют, выполняется запрос в базу данных.
4. Результат сериализуется в JSON и сохраняется в Redis с TTL.
5. Ответ возвращается клиенту.

### 3.1. Endpoint-ы с кешированием

| Endpoint                | Источник данных | Ключ в Redis              | TTL       |
| ----------------------- | --------------- | ------------------------- | --------- |
| `GET /v1/folders`       | PostgreSQL      | `cache:folders:pg:all`    | 60 секунд |
| `GET /v2/mongo/folders` | MongoDB         | `cache:folders:mongo:all` | 60 секунд |

### 3.2. Проверка работы кеширования

Работа кеширования была проверена вручную через `curl` и `redis-cli`.

После вызова endpoint-ов:

* `GET /v1/folders`
* `GET /v2/mongo/folders`

в Redis появились ключи:

* `cache:folders:pg:all`
* `cache:folders:mongo:all`

Также была подтверждена установка TTL:

* `TTL cache:folders:pg:all`
* `TTL cache:folders:mongo:all`

и наличие сериализованного JSON в кеше через команды `GET`.

Пример проверки:

```bash
docker exec -it file_storage_redis redis-cli
KEYS *
TTL cache:folders:pg:all
TTL cache:folders:mongo:all
GET cache:folders:mongo:all
```

### 3.3. Инвалидация кеша на практике

После успешного создания или удаления папки соответствующий ключ удаляется из Redis:

* PostgreSQL: `cache:folders:pg:all`
* MongoDB: `cache:folders:mongo:all`

Таким образом, следующий запрос повторно обращается к базе и формирует новый актуальный кеш.

## 4. Проектирование rate limiting

Rate limiting используется для защиты API от перегрузки и злоупотреблений.
В файловом сервисе наиболее критичными являются endpoint-ы, которые могут часто вызываться автоматически или использоваться для перебора данных.

### 4.1. Endpoint-ы, требующие ограничения

| Endpoint                       | Причина ограничения                                        | Уровень риска |
| ------------------------------ | ---------------------------------------------------------- | ------------- |
| `POST /v1/auth/login`          | возможен brute-force подбор пароля                         | высокий       |
| `POST /v1/users`               | возможна массовая регистрация пользователей                | средний       |
| `GET /v1/users/search`         | может создавать нагрузку на PostgreSQL при поиске по маске | средний       |
| `GET /v1/files/by-name/{name}` | часто вызывается и создаёт повторную нагрузку при чтении   | средний       |

В рамках лабораторной работы ограничение реализуется для endpoint:

* `POST /v1/auth/login`

### 4.2. Причины выбора endpoint-а авторизации

Этот endpoint выбран, потому что он:

* связан с авторизацией;
* обращается к PostgreSQL;
* наиболее чувствителен к большому числу повторных запросов;
* может использоваться для brute-force атаки.

### 4.3. Выбор алгоритма rate limiting

Для endpoint `POST /v1/auth/login` выбран алгоритм **Fixed Window Counter**.

Принцип работы:

1. Для клиента формируется Redis-ключ.
2. В Redis хранится счётчик запросов за фиксированное окно времени.
3. При каждом новом запросе счётчик увеличивается.
4. Если количество запросов превышает лимит, сервис возвращает ошибку `429 Too Many Requests`.
5. После завершения временного окна ключ автоматически удаляется за счёт TTL.

### 4.4. Обоснование выбора Fixed Window Counter

Алгоритм Fixed Window Counter выбран по следующим причинам:

* простая реализация;
* хорошо подходит для учебного проекта;
* не требует хранения истории всех запросов;
* потребляет мало памяти;
* легко реализуется через Redis `GET` + `SET` с TTL.

Недостатком алгоритма является возможность всплеска запросов на границе временных окон. Для текущей лабораторной работы этот недостаток допустим.

### 4.5. Лимиты запросов

| Endpoint              | Алгоритм             | Лимит                   | Ключ Redis                     | TTL       |
| --------------------- | -------------------- | ----------------------- | ------------------------------ | --------- |
| `POST /v1/auth/login` | Fixed Window Counter | 10 запросов / 60 секунд | `rate_limit:login:{client_ip}` | 60 секунд |

В реализованной версии для локального тестирования использовался ключ вида:

* `rate_limit:login:local-client`

### 4.6. Поведение при превышении лимита

Если клиент превышает лимит, сервис возвращает:

```http
HTTP/1.1 429 Too Many Requests
```

Также в ответ добавляются заголовки:

* `X-RateLimit-Limit`
* `X-RateLimit-Remaining`
* `X-RateLimit-Reset`

Где:

* `X-RateLimit-Limit` — максимальное число запросов в окне;
* `X-RateLimit-Remaining` — количество оставшихся запросов;
* `X-RateLimit-Reset` — время до сброса окна в секундах.

## 5. Реализация rate limiting

В рамках данного этапа реализован механизм ограничения количества запросов для endpoint:

* `POST /v1/auth/login`

### 5.1. Хранение состояния

Для хранения счётчиков используется Redis.
Ключ формируется по шаблону:

* `rate_limit:login:{client_ip}`

В процессе тестирования Redis Monitor показал реальные операции:

* `get rate_limit:login:local-client`
* `set rate_limit:login:local-client "1" "PX" "60000"`

Это подтверждает использование Redis в качестве хранилища состояния rate limiting.

### 5.2. Логика обработки запроса

При каждом запросе на логин выполняются следующие действия:

1. Определяется идентификатор клиента.
2. Формируется Redis-ключ rate limit.
3. Из Redis считывается текущее количество запросов.
4. Счётчик увеличивается на 1.
5. Обновлённое значение сохраняется в Redis с TTL 60 секунд.
6. Если количество запросов превышает лимит, возвращается `429 Too Many Requests`.
7. Если лимит не превышен, запрос на авторизацию обрабатывается дальше.

### 5.3. Установленные ограничения

Для endpoint `/v1/auth/login` установлено ограничение:

* **10 запросов в 60 секунд на одного клиента**

### 5.4. HTTP-заголовки

В ответах возвращаются следующие заголовки:

* `X-RateLimit-Limit: 10`
* `X-RateLimit-Remaining: <remaining>`
* `X-RateLimit-Reset: 60`

### 5.5. Проверка работы

Работа rate limiting была проверена с помощью серии последовательных запросов к endpoint авторизации.

Пример команды PowerShell:

```powershell
for ($i = 1; $i -le 12; $i++) {
  curl.exe -i -X POST "http://localhost:8080/v1/auth/login" `
    -H "Content-Type: application/json" `
    --data-binary "@login.json"
}
```

Результат проверки:

* первые 10 запросов — успешные (`HTTP 200 OK`);
* после исчерпания лимита приходят ответы:

  * `HTTP 429 Too Many Requests`
  * `{"error":"too many login requests"}`

Также в ответах корректно уменьшался заголовок:

* `X-RateLimit-Remaining`

до значения `0`.

### 5.6. Проверка через Redis MONITOR

Для дополнительной проверки использовалась команда:

```bash
docker exec -it file_storage_redis redis-cli MONITOR
```

В ходе выполнения запросов были зафиксированы команды:

```text
"get" "rate_limit:login:local-client"
"set" "rate_limit:login:local-client" "1" "PX" "60000"
"set" "rate_limit:login:local-client" "2" "PX" "60000"
"set" "rate_limit:login:local-client" "3" "PX" "60000"
```

Это подтверждает, что счётчик запросов действительно хранится в Redis и обновляется при каждом логине.

### 5.7. Вывод по реализации

Реализованный механизм rate limiting:

* защищает endpoint аутентификации от злоупотреблений;
* снижает риск brute-force атак;
* ограничивает влияние одного клиента на остальных пользователей;
* повышает стабильность API при всплесках нагрузки.

## 6. Анализ влияния механизмов производительности

В ходе лабораторной работы были внедрены два механизма повышения производительности системы:

* кеширование через Redis;
* rate limiting для endpoint-а авторизации.

### 6.1. Влияние кеширования

Кеширование позволяет:

* снизить количество обращений к PostgreSQL и MongoDB;
* уменьшить время отклика при повторяющихся чтениях;
* повысить пропускную способность системы.

В текущем сервисе особенно эффективно кеширование применяется к операциям получения списка папок.

### 6.2. Влияние rate limiting

Rate limiting:

* ограничивает количество запросов от одного клиента;
* предотвращает перегрузку endpoint-а авторизации;
* защищает систему от автоматических повторных запросов;
* стабилизирует нагрузку на PostgreSQL.

### 6.3. Полезные метрики для мониторинга

| Метрика            | Назначение                           |
| ------------------ | ------------------------------------ |
| Response Time      | время обработки запроса              |
| Throughput         | количество запросов в секунду        |
| Cache Hit Rate     | доля обращений, обслуженных из кеша  |
| Rate Limit Rejects | число отклонённых запросов по лимиту |

### 6.4. Оценка эффективности кеширования

Эффективность кеширования может оцениваться через показатель **Cache Hit Rate**:

```text
Hit Rate = Hits / (Hits + Misses)
```

где:

* `Hits` — запросы, обслуженные из Redis;
* `Misses` — запросы, приведшие к обращению в базу данных.

Для базовой оценки можно использовать:

* Redis `INFO stats`
* анализ ключей и TTL
* логирование попаданий и промахов на уровне приложения

### 6.5. Общий вывод

Добавление Redis в сервис позволило расширить архитектуру приложения без переработки основной бизнес-логики.
Кеширование уменьшает нагрузку на базы данных при повторяющихся чтениях, а rate limiting делает сервис более устойчивым к злоупотреблениям и резким всплескам запросов.

```bash
    --data-binary "@login.json"
done
```

Ожидаемый результат:

* сначала `HTTP 200 OK`;
* после исчерпания лимита — `HTTP 429 Too Many Requests`.
