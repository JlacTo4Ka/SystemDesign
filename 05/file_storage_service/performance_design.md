1. Анализ производительности

В рамках данной лабораторной работы рассматривается уже существующий сервис file_storage_service, в котором реализованы операции работы с пользователями, папками и файлами.
Так как лабораторная работа является продолжением предыдущих, задача заключается не в переработке всей архитектуры, а в добавлении механизмов кеширования и rate limiting поверх текущего проекта.

Hot paths

Наиболее часто используемыми операциями в файловом сервисе являются операции чтения данных и авторизации. К основным hot path можно отнести следующие endpoint-ы:

```
POST /v1/auth/login
GET /v1/folders
GET /v1/files/... получение файла по имени
GET /v2/mongo/folders
```

Эти операции являются наиболее важными, так как напрямую связаны с базовыми действиями пользователя в системе: входом в систему, просмотром списка папок и получением информации о файлах.

Потенциально медленные операции

Наиболее затратными по времени могут быть операции, требующие обращения к базе данных:

проверка логина и пароля в PostgreSQL при авторизации
получение списка папок из PostgreSQL
получение списка папок из MongoDB
поиск файла по имени

При низкой нагрузке эти операции могут выполняться быстро, однако при большом количестве повторяющихся запросов они начинают создавать лишнюю нагрузку на базы данных и увеличивать общее время ответа сервиса.

Требования к производительности

Для сервиса можно принять следующие ориентиры по производительности:

простые операции чтения должны выполняться в пределах 100–150 мс
endpoint авторизации должен отвечать в пределах 150–200 мс
повторяющиеся запросы не должны каждый раз обращаться к базе данных
один клиент не должен иметь возможность перегрузить endpoint авторизации большим числом запросов
Вывод по анализу

Наиболее подходящими кандидатами для кеширования в данном сервисе являются endpoint-ы получения списков папок, так как они относятся к часто читаемым данным и удобны для инвалидации при изменении состояния системы.

Для реализации кеширования были выбраны:

GET /v1/folders
GET /v2/mongo/folders

Такой выбор позволяет уменьшить нагрузку как на PostgreSQL, так и на MongoDB.

Для rate limiting был выбран endpoint:

POST /v1/auth/login

Это связано с тем, что endpoint авторизации наиболее чувствителен к большому количеству повторных запросов и требует защиты от brute-force и избыточной нагрузки.

2. Проектирование стратегии кеширования

Для повышения производительности сервиса было принято решение добавить кеширование поверх уже существующей логики работы с PostgreSQL и MongoDB.
Так как лабораторная работа является продолжением предыдущего проекта, кеширование внедряется в текущую архитектуру без переработки основной бизнес-логики.

Выбор данных для кеширования

В качестве кандидатов для кеширования были выбраны часто читаемые данные, которые:

регулярно запрашиваются клиентом;
не изменяются при каждом запросе;
могут безопасно храниться ограниченное время;
легко инвалидируются при изменении состояния системы.

Для реализации кеширования были выбраны следующие endpoint-ы:

GET /v1/folders
GET /v2/mongo/folders
Обоснование выбора

Endpoint получения списка папок является одним из основных в файловом сервисе, так как при работе пользователя список папок отображается регулярно.
Повторные обращения к этим endpoint-ам без кеширования каждый раз приводят к чтению данных из базы.

Выбор именно этих двух endpoint-ов позволяет:

уменьшить нагрузку на PostgreSQL;
уменьшить нагрузку на MongoDB;
сократить время ответа для повторяющихся запросов;
показать кеширование для двух разных backend-ов в рамках одного сервиса.
Выбранная стратегия кеширования

Для обоих endpoint-ов выбрана стратегия Cache-Aside.

Принцип работы следующий:

При поступлении запроса сервис сначала проверяет наличие данных в Redis.
Если данные найдены, они возвращаются из кеша.
Если данных нет, сервис получает данные из базы данных.
После этого данные сохраняются в Redis с установленным временем жизни.
Ответ возвращается клиенту.
Причины выбора Cache-Aside

Стратегия Cache-Aside выбрана по следующим причинам:

простая реализация;
хорошо подходит для учебного проекта;
не требует изменения логики работы базы данных;
позволяет гибко контролировать время жизни кеша;
легко комбинируется с существующими PostgreSQL и MongoDB endpoint-ами.
Время жизни кеша (TTL)

Для кешируемых данных устанавливается TTL 60 секунд.

Такой TTL выбран как компромисс между:

уменьшением нагрузки на базу данных;
сохранением достаточно актуальных данных;
простотой реализации.

Для данной лабораторной работы значение 60 секунд является достаточным, так как список папок не изменяется настолько часто, чтобы требовалась более агрессивная стратегия обновления.

Ключи кеша

Для хранения данных в Redis будут использоваться следующие ключи:

cache:folders:pg:all — список папок из PostgreSQL
cache:folders:mongo:all — список папок из MongoDB
Инвалидация кеша

Кеш должен сбрасываться при изменении данных, влияющих на содержимое списка папок.

Для PostgreSQL-ручек инвалидация выполняется при:

POST /v1/folders
DELETE /v1/folders/{id}

Для MongoDB-ручек инвалидация выполняется при:

POST /v2/mongo/folders
DELETE /v2/mongo/folders/{id}

Таким образом, после создания или удаления папки соответствующий кеш удаляется, и следующий запрос снова получит актуальные данные из базы.

Ожидаемый эффект

Ожидается, что использование кеширования позволит:

сократить количество повторных обращений к PostgreSQL и MongoDB;
уменьшить среднее время ответа для endpoint-ов получения списков папок;
повысить пропускную способность сервиса;
уменьшить нагрузку на backend при повторяющихся запросах.
